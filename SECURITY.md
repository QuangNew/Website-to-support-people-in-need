# Security Policy

## Supported versions

This project is actively maintained on the `main` branch and the latest deployed release. Older branches or historical snapshots should be treated as unsupported unless explicitly stated otherwise.

| Version | Supported |
| ------- | --------- |
| `main` | Yes |
| Latest release | Yes |
| Older branches/releases | No |

## Reporting a vulnerability

Please do not report security vulnerabilities through public GitHub issues, pull requests, or discussions.

Use one of these private channels instead:
1. GitHub Security Advisories / the repository's private vulnerability reporting flow, if enabled.
2. Private contact with the repository maintainer through GitHub.

If GitHub Security, Dependabot alerts, or Dependabot security update pull requests already surface the issue, continue through those private security workflows instead of opening a public issue.

When reporting, please include:
- A clear summary of the issue
- Affected area or file paths
- Reproduction steps or proof of concept
- Estimated impact
- Any suggested mitigation, if available

## What to expect

Maintainers will review reports as quickly as possible and aim to:
- Confirm whether the issue is valid
- Assess severity and impact
- Prepare a fix or mitigation
- Coordinate public disclosure after a fix is available, when appropriate

## Disclosure guidance

Please give maintainers reasonable time to investigate and patch the issue before sharing full details publicly.

## Scope reminders

Useful reports typically involve issues such as:
- Authentication or authorization bypass
- Sensitive data exposure
- Injection vulnerabilities
- Broken access control
- File upload or deserialization issues
- Dependency vulnerabilities with real impact on this project

Non-security bugs, feature requests, and documentation improvements should go through the normal issue or pull request process.
